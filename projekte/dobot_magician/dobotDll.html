Warum diese beiden Dateien wichtig sind

Um den Dobot Magician aus Python zu steuern, werden zwei Ebenen benoetigt: die native Windows-Bibliothek DobotDll.dll und die Python-Datei DobotDllType.py, die diese Bibliothek kapselt. 

dein Python-Programm
    -> DobotDllType.py
    -> DobotDll.dll
    -> USB / serielle Verbindung
    -> Dobot Magician

Kurz gesagt: DobotDll.dll spricht mit der Hardware, DobotDllType.py macht diese Funktionen in Python nutzbar.

Aufgabe der einzelnen Dateien

DobotDll.dll

  • Native Bibliothek (Windows-DLL) fuer den Zugriff auf den Roboter.
  • Wird zur Laufzeit durch Python geladen.
  • Muss zur Python-Architektur passen (32-Bit oder 64-Bit).

DobotDllType.py

  • Python-Wrapper um die Funktionen der DLL.
  • Enthaelt Datentypen, Konstanten, Modi und API-Funktionen.
  • Wird im Code meist als import DobotDllType as dType eingebunden.

Empfohlene Ordnerstruktur

In unseren Projekten hat sich folgende Struktur bewaehrt:

Dobot-Projekt/
|- mein_programm.py
|- sdk64/
|  |- DobotDllType.py
|  |- DobotDll.dll
|- sdk32/
   |- DobotDllType.py
   |- DobotDll.dll
64-Bit-Python benoetigt die 64-Bit-DLL. 32-Bit-Python benoetigt die 32-Bit-DLL. Eine gemischte Kombination fuehrt typischerweise zu Ladefehlern.

Minimales Startbeispiel (Windows, 64-Bit)

Das folgende Beispiel registriert den DLL-Ordner sauber und baut danach die Verbindung auf. 

from pathlib import Path
import os
from sdk64 import DobotDllType as dType

sdk_verzeichnis = Path(__file__).resolve().parent / "sdk64"
os.add_dll_directory(str(sdk_verzeichnis))

api = dType.load()
zustand = dType.ConnectDobot(api, "", 115200)

if zustand[0] == dType.DobotConnect.DobotConnect_NoError:
    print("Dobot verbunden")
else:
    print("Verbindung fehlgeschlagen:", zustand)

dType.DisconnectDobot(api)

Bei dieser Variante kann DobotDll.dll in sdk64 liegen, ohne dass sie im Systempfad eingetragen werden muss.

Typische Fehler und schnelle Pruefung

Fehlerbild

  • FileNotFoundError: Could not find module 'DobotDll.dll'
  • Import von DobotDllType.py klappt, Laden der DLL aber nicht.
  • Verbindung wird nicht hergestellt, obwohl COM-Port vorhanden ist.

Pruefpunkte

  • Passt 32/64-Bit von Python und DLL zusammen?
  • Liegt die DLL wirklich im erwarteten Ordner?
  • Wurde der DLL-Ordner mit os.add_dll_directory() registriert?
  • Ist der Roboter eingeschaltet und per USB korrekt verbunden?
Gute Praxis: zuerst nur Laden und Verbinden testen, danach erst Bewegungsbefehle einbauen.
